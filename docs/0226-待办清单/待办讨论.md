# CodexRelay 待办讨论（2026-02-26）

更新：2026-02-28（本文件会随着实现进度持续更新）

目标：做一个 macOS / Windows / Linux 通用的 GUI 工具，用于“导出/传递/导入” Codex CLI session（`rollout-*.jsonl`），并且在本机永久保存每一次传递的版本（vault + SQLite 历史记录），支持冲突/分叉时安全续接；支持“更换 session id”。

本文把需求拆成可落地的里程碑与详细待办清单，便于我们按优先级逐步实现。

---

## 已确认决策（2026-02-26）

- Bundle 形式：zip
- Vault：同时保存 B（bundle.zip）和 A（解包后的 manifest/rollout 等）
  - 先做 B（实现最简单、也最贴近“传递实体”）
  - 同时补 A（方便可视化/检索/预览）
- 导入冲突默认策略：同 `session_id` 且 sha256 不同 -> 推荐“改 ID 导入”（保留分叉，两条都可 resume）
- 更换 session id：默认不改 `forked_from_id`（允许用户在高级选项里手动改）
- UI：需要明确区分/提示“检测到的 Codex 数据目录”（以及是否来自环境变量/默认探测）

## 0. 关键约束（不要踩坑）

- 原生兼容的前提：导入后目标机本地必须出现对应的 `CODEX_HOME/sessions/**/rollout-*.jsonl`，且其中 `session_meta.payload.id == session_id`，这样 `codex resume <session_id>` 才能工作。
- 不要在 bundle 中携带 `CODEX_HOME/auth.json` / `config.toml`（敏感信息，且非必须）。
- 同一个 session 在两台机器“同时续写”会分叉，JSONL 不能可靠自动合并；正确做法是“强制备份 + 让用户选择覆盖或改 ID 导入”。
- `shell_snapshots/<id>.sh` 可能包含环境变量/路径等信息：默认不自动打包；如果需要，作为可选项并提示风险。
- 所有导入/导出都要做到“可恢复”：先落 vault 再写入 `CODEX_HOME`（即使写入失败也不丢）。

## 1. 待确认的问题（先定规则，避免返工）

1) Bundle 形式：zip（已确认）
2) Vault 存储策略：
   - A: vault 里存“原始 rollout.jsonl + manifest”（可读、可 diff）
   - B: vault 里存“完整 bundle.zip”（更接近传输实体）
   - 已确认：两者都存；先做 B，同时补 A
3) 冲突策略默认项：
   - 已确认：导入时若本机存在同 `session_id` 且 sha256 不同：默认推荐“改 ID 导入”
4) “更换 session id”是否同时改 `forked_from_id`？
   - 已确认：默认不改 `forked_from_id`；允许用户主动修改
5) Windows 端 Codex 数据目录：
   - 原生：`%USERPROFILE%\\.codex`
   - WSL：`~/.codex`（Linux 路径）
   - 已确认：需要在 UI 中区分/提示检测到的 Codex 数据目录（设置页里展示探测到的 CODEX_HOME）

## 3. 详细待办清单（按优先级）

说明：
- P0：必须；不做就无法闭环或风险很高
- P1：强烈建议；显著提升可用性/安全性
- P2：增强；可延后

### 3.1 Session 扫描与展示

- [x] [P0] Session 列表信息补全
  - 从 `rollout-*.jsonl` 提取：
    - `session_id`（payload.id）
    - `cwd`、`cli_version`、`model_provider`
    - `last_event_timestamp`（读取最后一行 JSON 的 `timestamp` 字段；不需要全文件解析）
    - `file_size`（sha256 作为 P1 可选项）
  - 验收：列表能显示“更新时间/会话ID/cwd/cli版本/Provider/最后事件时间”。

- [x] [P0] CODEX_HOME 探测与展示
  - 优先环境变量 `CODEX_HOME`，否则用 HOME/USERPROFILE + `/.codex`
  - 验收：首页显示当前 CODEX_HOME；若目录不存在则有提示。

- [x] [P1] Session 过滤/搜索
  - 按 session id 前缀、cwd 关键字过滤
  - 验收：能快速定位目标 session。

- [ ] [P1] Session 列表显示 sha256（可选）
  - 选择（2026-02-27）：本项目把 sha256 作为“版本号/完整性校验”的权威来源（导出/导入/历史记录），但不会在 Session 列表页默认全量计算（避免对大文件/大量会话造成卡顿）。
  - 可选实现方向：按需计算（仅选中会话/手动触发）+ 缓存（path+mtime+size），或做采样指纹用于快速提示“可能变化”。

### 3.2 Bundle 导出（Export）

- [x] [P0] 导出流程（强制命名）
  - 选择一个 session（或输入 session id）
  - 强制输入 `name`（可选 note/tag）
  - 生成 zip：
    - `manifest.json`
    - `rollout.jsonl`（从源路径复制）
    - 可选 `shell_snapshot.sh`
  - 验收：导出的 zip 能在另一台机器成功导入并 `codex resume <id>`。

- [x] [P0] manifest.json 生成与校验
  - 最少字段：
    - schema_version、name、session_id、created_at
    - source_device（生成本机 device_id）
    - files.sha256/size（至少 rollout）
  - 验收：导入时可以校验 sha256，发现损坏给出明确错误。

- [x] [P1] 导出前校验
  - 校验 rollout 第一行是 `session_meta`，且 `payload.id == session_id`（不一致则拒绝导出）
  - 验收：异常文件不会被误导出。

- [ ] [P2] 导出“仅备份到 vault”（不生成 zip）
  - 给高级用户用（本机只做版本管理）

### 3.3 Bundle 导入（Import）

- [x] [P0] 导入流程（解包 + 校验 + 写入）
  - 选择 zip
  - 校验 manifest schema_version、sha256、session_id 与 rollout 第一行 payload.id 一致
  - 将 bundle 先保存到 vault（保证可恢复）
  - 将 rollout 写入到 `CODEX_HOME/sessions/<YYYY>/<MM>/<DD>/...`：
    - 最简单：保持 rollout 原文件名与相对路径（如果 bundle 不含相对路径，就采用“按导入时当前时间写入当天目录 + 生成文件名”）
  - 验收：导入后在终端直接 `codex resume <session_id>` 成功。

- [x] [P0] 同 ID 冲突检测（sha256 比较）
  - 若本机已存在同 id：
    - 比较 sha256
    - 相同：提示“已存在相同版本”，允许仅写历史记录/跳过写入
    - 不同：进入冲突向导（M3 详述），至少要“自动备份本机版本到 vault”
  - 验收：不会因为覆盖导致不可逆丢失。

- [x] [P1] 导入后快捷操作
  - 一键复制命令：`codex resume <id>`（或显示按钮）
  - 一键打开文件夹：打开对应 rollout 所在目录（可用 tauri opener）

### 3.4 Vault 与历史记录（SQLite）

- [x] [P0] Vault 目录与基础写入
  - 位置：AppData/CodexRelay/vault（跨平台）
  - 结构：每次操作一个 transfer 目录：`vault/<transfer_id>/...`
    - 导出/改ID：会生成一个可传输 zip（文件名包含 op + 时间 + session_id + name）
    - 导入/恢复：会把用户选择的 zip 固定存为 `bundle.zip`，并同时解包出 `manifest.json` / `rollout.jsonl`
  - 验收：导入/导出/恢复/改ID都会在 vault 留档（不会自动删除）。

- [x] [P0] SQLite schema（MVP）
  - 表：`transfers`
    - id（uuid）
    - created_at
    - op（export/import/restore/change_id）
    - name（强制）
    - note（可选）
    - tags / favorite / updated_at（可选元信息，用于管理）
    - session_id_old / session_id_new（改 ID 或冲突导入时用）
    - effective_session_id（最终可 resume 的会话ID）
    - vault_dir / bundle_path / vault_rollout_rel_path（vault 留档位置）
    - rollout_sha256 / size
    - status（ok/failed + error_message）
    - local_rollout_path（本机写入/覆盖的目标文件路径，便于定位）
  - 验收：历史页能按时间列出所有记录，点击可查看详情。

- [x] [P1] 恢复/回滚（Restore）
  - 从 vault 选择某个历史版本 -> 写回 CODEX_HOME 对应 session（必要时覆盖并再次备份当前）
  - 验收：用户能把 session 恢复到任意历史版本并继续 resume。

- [x] [P1] 手动清理
  - UI 支持删除：
    - 单条历史记录
    - 同时删除 vault 文件（需要二次确认）
    - 多选/全选批量删除（可选同时删除 vault 文件，失败会返回错误列表）
  - 验收：不会自动删除；仅用户手动操作才删除。

### 3.5 冲突向导（M3 核心）

- [x] [P0] 冲突向导 UI（导入时触发）
  - 展示两份版本的摘要：
    - 本机版本：mtime、sha256、最后事件时间、cwd
    - 导入版本：manifest.created_at、sha256、最后事件时间、source_device
  - 选项：
    1) 覆盖本机（已自动备份本机版本到 vault）
    2) 改 ID 导入为新会话（推荐）
    3) 取消（不写入 CODEX_HOME，但仍保留 vault+历史记录）
  - 验收：每个分支都不会丢版本。

### 3.6 更换 Session ID（Change ID）

- [x] [P0] 生成新 session id
  - 已决定：默认生成 UUID v7（与 Codex 现有 id 风格一致）
  - 验收：生成的 id 能被 `codex resume <id>` 正常识别（需要实测）。

- [x] [P0] JSONL 精确重写（避免误替换）
  - 逐行 JSON parse（只处理合法 JSON 行）
  - 仅修改：
    - `type == "session_meta"` 且 `payload.id == old_id` -> new_id
  - 写出到新的 rollout 文件名（包含 new_id）；不修改原文件
  - 验收：改 ID 后导入的会话可以 resume；不会误改用户对话文本里出现的 id。

- [ ] [P1] 选择性处理 fork 链
  - UI 选项：
    - 保留 `forked_from_id`
    - 清空 `forked_from_id`
  - 验收：两种方式都能 resume（仅影响显示/链路）。

### 3.7 设置与设备信息

- [x] [P1] 设备身份（device_id）
  - 首次启动生成并保存到 SQLite（kv 表）
  - 展示 hostname、os、arch（用于 manifest.source_device）
  - 验收：manifest 中能追溯来源设备。

- [x] [P1] CODEX_HOME 设置页

  - 已补充：Vault 使用量统计（按 transfer 目录扫描，便于清理）。
